Skema Integrasi Pembayaran Virtual Account & Transfer Otomatis
Ammu Online  ⇄  Sistem BMT PETA

Acuan teknis-konseptual untuk dipelajari BMT PETA Pusat, tim pengembang sistem BMT, dan General Manager — menghubungkan tagihan santri di aplikasi Ammu Online dengan kanal pembayaran BMT PETA secara otomatis tanpa upload bukti manual, sekaligus integrasi dua arah: menarik pembayaran santri lewat Virtual Account (masuk) dan menyalurkan bisyaroh guru ke rekening BMT (keluar).

Pondok Pesantren Mambaul Ulum Aplikasi: Ammu Online Juni 2026 · v.1 draft Status: Usulan — menunggu detail API BMT

1.Ringkasan Eksekutif

Apa yang diusulkan dan kenapa.

Ammu Online adalah sistem manajemen pesantren (data santri, akademik, dan keuangan) yang dipakai Pondok Mambaul Ulum. Modul keuangannya sudah menghasilkan tagihan per santri (syahriyah, infaq, iuran). Saat ini pembayaran non-tunai dilakukan lewat transfer bank biasa lalu wali meng-upload bukti dan admin memverifikasi manual.

Usulannya: mengganti proses manual itu dengan Virtual Account (VA) dari BMT PETA. Setiap santri memperoleh satu nomor VA tetap. Wali cukup membayar ke nomor tersebut melalui kanal BMT/bank, dan sistem BMT memberi konfirmasi otomatis ke Ammu Online sehingga tagihan langsung tertandai LUNAS — tanpa upload bukti, tanpa verifikasi manual.

Inti yang diharapkan dari BMT PETA Satu hal yang membuat seluruh skema ini bisa berjalan otomatis adalah kanal konfirmasi pembayaran dari sisi BMT (notifikasi/“webhook”, atau API status yang bisa ditanya Ammu). Tanpa sumber konfirmasi dari BMT, pembayaran tidak bisa ditandai lunas otomatis. Bagian ini sengaja kami tonjolkan agar bisa didiskusikan dengan tim pengembang BMT.

3.Model Virtual Account: Tetap per Santri

Pilihan yang diambil: setiap santri punya satu nomor VA permanen.

Nomor VA dibentuk dari kode institusi BMT (prefix yang ditetapkan BMT) digabung dengan identitas santri (NIS/ID santri). Nomor ini tetap — dipakai untuk semua tagihan santri tersebut (syahriyah bulanan, infaq, iuran), berapa pun nominalnya.

PropertiKeterangan
JenisVA tetap/permanen (bukan per-tagihan) — satu nomor seumur santri aktif
Pembentukprefix_BMT + NIS · disimpan di data santri sebagai va_bmt
NominalTerbuka — wali bisa bayar sesuai tagihan (open amount), atau BMT bisa menutup ke nominal tagihan bila didukung
IdentifikasiSaat ada pembayaran masuk, BMT mengirim nomor VA → Ammu tahu ini santri siapa
KelebihanMudah diingat & dibagikan ke wali, tidak perlu generate ulang tiap bulan
CatatanKarena open amount, pencocokan ke tagihan tertentu dilakukan di sisi Ammu (alokasi ke tagihan terdekat / sesuai instruksi wali)

Tetap aman saat transisi Alur lama (transfer + upload bukti) tidak dihapus — dipertahankan sebagai cadangan sampai integrasi BMT benar-benar siap dan teruji. VA diaktifkan lewat sebuah saklar pengaturan, sehingga peralihan bisa bertahap dan aman.

6.Opsi Mekanisme Konfirmasi

Tiga cara Ammu bisa tahu pembayaran sudah masuk — dipilih bersama tim BMT.

★ Rekomendasi

A. Webhook (push) dari BMT

Tiap ada pembayaran VA masuk, sistem BMT otomatis memanggil endpoint Ammu (mengirim nomor VA, nominal, waktu, ID transaksi).

+ Real-time, paling hemat, paling rapi
− Perlu BMT mendukung webhook keluar + tanda tangan keamanan
Alternatif

B. Polling API BMT

Ammu berkala (mis. tiap beberapa menit) menanyakan status pembayaran ke API BMT.

+ Tidak butuh webhook; cukup API status
− Tidak instan; beban kueri berkala
Alternatif

C. Rekonsiliasi BMT

BMT yang menandai lunas di sistemnya; Ammu menampilkan/menarik data dari BMT (mis. file/laporan harian).

+ Cocok bila keuangan dipusatkan di BMT
− Paling lambat; bergantung jadwal rekon
Keputusan menunggu BMT Mekanisme final mengikuti kemampuan sistem BMT PETA. Ammu Online dirancang fleksibel sehingga bisa menyesuaikan ke salah satu dari tiga opsi di atas tanpa merombak aplikasi.

9.Keamanan & Tata Kelola

  • Tanda tangan data (HMAC/signature) pada setiap konfirmasi — memastikan notifikasi benar-benar dari BMT, bukan pihak lain.
  • Anti-dobel (idempoten) — satu bmt_trx_id hanya diproses sekali, mencegah tagihan ter-lunas ganda.
  • Tanpa menyimpan data kartu/PIN — Ammu hanya menerima status pembayaran, tidak menyentuh kredensial perbankan wali.
  • Jejak audit — setiap pelunasan otomatis tercatat (sumber, waktu, ID transaksi) di Buku Induk & log audit yang sudah ada di Ammu.
  • Kanal terenkripsi (HTTPS) untuk semua komunikasi.

10.Tahapan Implementasi

Bertahap, dengan alur lama sebagai jaring pengaman.

1

Kesepakatan & spesifikasi

BMT & pondok sepakati mekanisme (§6), format VA, dan akses sandbox. butuh BMT

2

Siapkan kerangka di Ammu (sudah dimulai)

Opsi VA & konfigurasi BMT disiapkan di aplikasi dalam keadaan non-aktif (saklar). Tidak mengganggu operasi berjalan.

3

Integrasi & uji sandbox

Sambungkan ke API/webhook BMT di lingkungan uji, uji pembayaran & konfirmasi end-to-end.

4

Uji terbatas (pilot)

Aktifkan untuk sebagian santri; alur lama tetap tersedia sebagai cadangan.

5

Live penuh

Aktifkan untuk semua santri setelah stabil; upload bukti manual menjadi opsi darurat saja.

11.Integrasi Dua Arah: Pencairan Bisyaroh Guru

BMT tidak hanya menarik pembayaran santri (masuk), tapi juga menyalurkan bisyaroh ke guru (keluar).

Semua guru/pegawai sudah punya rekening BMT, dan bisyaroh selama ini ditarik dari sana. Ammu Online sudah menghitung bisyaroh tiap guru (pokok + bonus kehadiran + tunjangan − potongan). Usulannya: hasil hitungan itu dipakai untuk mencairkan ke rekening BMT guru, dan begitu pencairan dikonfirmasi, otomatis tercatat sebagai kas keluar di Buku Induk pondok — menutup celah pencatatan yang sekarang masih manual.

Tiga model eksekusi pencairan

★ Target pondok

Model B — API BMT (full otomatis)

Ammu kirim perintah pencairan → BMT transfer pondok→guru sendiri → Ammu catat dari status BMT.

+ Benar-benar otomatis
− Butuh API disbursement dari BMT
Cadangan

Model A — Admin transfer, app catat

Admin transfer via BMT, lalu konfirmasi di Ammu → otomatis tercatat kas keluar + slip ditandai cair.

+ Jalan tanpa API, bisa sekarang
− Transfer masih manual
Jalan tengah

Model C — File pencairan massal

Ammu ekspor file daftar (format BMT) → admin unggah ke sistem BMT untuk eksekusi sekaligus.

+ Satu unggah banyak guru
− Perlu fitur bulk-upload BMT
Keputusan pondok Target Model B (penuh otomatis via API BMT), dengan Model A sebagai cadangan bila BMT belum memiliki API disbursement. Persetujuan pencairan dilakukan oleh admin keuangan.

Yang dibutuhkan dari BMT (sisi pencairan)
